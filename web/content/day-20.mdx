### The three-leg mental model

- **GitHub** holds the truth.
- **Vercel** mirrors `main` to a URL on every push.
- **Supabase** is the database your URL talks to over HTTPS.
- Every `git push` to `main` redeploys. That's the loop.

### What OpenCode is doing for you

- `opencode /init` reads your repo and writes a CLAUDE.md / AGENTS.md stub.
- `opencode /run "add a page that reads my supabase 'posts' table"` writes the page, the type, the env wiring — and asks before touching anything sensitive.
- You're the **director**. The agent is the typist.

## 🧪 Lab: One page, one URL, one row

1. `npx create-next-app@latest capstone --ts --tailwind --app` (or use your existing prototype scaffold).
2. `git init && git add . && git commit -m "deploy: bootstrap"`. Push to a new GitHub repo.
3. vercel.com → New Project → import repo → deploy. Wait for the green check.
4. supabase.com → New project → Table editor → create `posts (id uuid, body text)`. Insert one row.
5. Copy Supabase Project URL + anon key into `.env.local` *and* Vercel → Settings → Environment Variables.
6. Run `opencode /run "make the home page show the posts.body from supabase"`. Review the diff, accept.
7. Push. Watch Vercel redeploy. Open URL.

**Artifact.** Live URL + GitHub repo + screenshot of OpenCode's diff that you accepted.

> ⚠️ **Indian-context gotcha.** Campus Wi-Fi blocks Vercel CLI auth on some networks. If `vercel login` hangs, hotspot for 60s.
