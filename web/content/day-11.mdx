> 🧠 **Where this fits.** You've spent four days on design thinking — problem, user, idea, features. Today bridges that work into the building weeks ahead. Before you wire up any AI feature, you need to know *what data feeds it* and *what decision comes out the other side*.

## 🎯 Today's objective

**Topic.** Identify intelligent data inside your capstone, decide what to capture and what to build from it.

**By end of class you will have:**
1. A list of the **raw signals** your product can capture (clicks, text, time, location, ratings, etc.).
2. For each, the **intelligent signal** you can derive (intent, sentiment, anomaly, segment, recommendation).
3. For each intelligent signal, the **decision** it unlocks for the user or the system.

> *Why this matters.* AI features without data plans become demoware. The teams that ship next month already know — today — exactly what their app captures on day 1 and what it predicts on day 30.

> 🧠 **Quick glossary.** **Raw data** = what happened (a row in a log). **Intelligent data** = what it means (a label, score, prediction). **Decision** = what changes because of the intelligent signal. **Ground truth** = the answer you'd compare a prediction against. **Cold start** = the problem of having no data on day 1.

### The three layers of data

1. **Raw.** What the system captured. A click, a rating, a search query, a GPS ping, a message, a transaction.
2. **Intelligent.** What you derived from the raw. *This user is at risk of churning. This review is sarcastic. These two products are substitutes. This transaction is anomalous.*
3. **Decision.** What the product does differently because of the intelligent signal. *Send a discount. Hide the review. Bundle the products. Block the transaction.*

If a capstone idea can't fill all three columns, it's not yet a product — it's a feature looking for a problem.

### Where to *find* data when you have none

- **First-party.** Instrument your own app from day 1. Even 50 rows of real user behaviour beats 50,000 rows of synthetic.
- **Third-party APIs.** Google Maps, OpenWeather, public Twitter/X, GitHub — what your product can read on demand.
- **Public datasets.** Kaggle, Hugging Face, Google Dataset Search, government data portals (data.gov.in for India).
- **Synthetic / LLM-generated.** Useful for cold start but never for evaluation.

### What "intelligent" actually means (taxonomy)

- **Classification.** Is this review positive? Is this transaction fraud?
- **Regression.** What rating will this user give? How long until churn?
- **Ranking / recommendation.** Out of 10,000 items, which 5 first?
- **Clustering / segmentation.** Which users behave alike?
- **Anomaly detection.** What's different this hour vs every other hour?
- **Generation.** Given context, produce a reply / image / summary.

## 🧪 Lab: Raw → Intelligent → Decision map

Open a Google Sheet (or Notion table). Three columns:

| Raw signal | Intelligent signal | Decision unlocked |
|---|---|---|
| Search query text | Inferred intent (browse / buy / help) | Different home screen layout |
| GPS pings every 5 min | Commute pattern | Suggest evening route |
| Star rating + review text | Sarcasm flag | Don't auto-publish 5-star sarcastic reviews |

1. **Fill 5 rows** for your capstone. Be honest — don't write rows you don't actually need.
2. For each intelligent signal, **name how you'd build it.** LLM prompt? Embedding similarity? Simple rule? Classifier?
3. For each, **estimate cold-start.** Can you ship a v1 with 0 users of data? 100? 10,000?
4. **Mark one row as risky** — the intelligent signal that, if you got it wrong, would hurt the user most.

**Artifact.** Submit your sheet on the dashboard before Day 12. ~5 rows, all three columns filled.
