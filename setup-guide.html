Support Faculty — Windows lab computers

Lab setup guide (Windows)

Written for non-technical staff: official download links, normal Windows install wizards, and checklists. Mac/Linux and command-line flows are not on this page—ask Gnanalytica or college IT if you need that. Students do not install software during class; get lab PCs ready before Day 1.

1. Before you start

You need Administrator (admin) rights on every PC. If the installer never asks for permission, or you cannot click “Yes” on a Windows security prompt, stop and get IT to sign in, or do setup on a day an admin is present.

What to plan for

  • About 45–60 minutes per computer the first time (faster on good campus Wi-Fi or a wired port)
  • Google Chrome as the main browser (this workshop is run and tested in Chrome)
  • Enough space for big AI downloads—see System & network (about 40 GB free is a comfortable target)

Shared college PCs: “all users” when the installer offers it

If more than one student will log in with a different Windows account, use options like “install for all users of this computer” or “system” if you see them. “For me / current user only” is risky on shared lab PCs—if that is the only option, check with IT.

Students and home laptops

In the classroom students are not expected to install software. On a home laptop (if your cohort allows it), you can point people at this same page to repeat the same Windows steps in their own time.

AI model downloads (Ollama) are large (often several GB). Finish that on a calm day, not 10 minutes before a session. See Ollama below.

1A. One pass through everything (checklist)

Work top to bottom. You do not need a terminal: use the numbered links in section 3, the browser for accounts, and the “check it works” list at the end. Tick boxes as you go (they reset if you refresh the page—print with Ctrl + P if you want a paper list).

Target: complete this at least a few days before the cohort’s first in-person lab, so there is time to get help.

2. System & network (Windows)

Many college labs are already a good match: 64-bit Windows, 16 GB RAM, 512 GB SSD, and a recent Intel/AMD i5. That is enough to run the workshop, Cursor, and the smaller on-machine AI models.

You do not need to type this list into the PC yourself—send it to the network/IT helpdesk. Campus filters often block one of these, which then breaks a single day of the course.

Still “cannot sign in / cannot download” on day one? It is often the firewall, not a bad password. Ask IT to test one machine against the list above.

2B. Sign-ins and keys (all in the browser, plus Windows settings)

A working install is not enough: most “it worked yesterday” issues are accounts. In Chrome, on one lab machine, walk through the sign-ins your trainer gave you—often including Google, GitHub, Groq, Vercel, Supabase, Hugging Face, and a few more. You do that by opening each product’s normal “Sign in / Log in” page and using your college or cohort email, not by typing secrets into the workshop chat.

  • Google account — needed for many AI tools. Use the account your cohort was told to use, or the college G Suite policy.
  • GitHub — github.com/signup if the college does not already provide accounts.
  • API keys (Gemini, Groq, and others) — the trainer or IT will tell you where to paste them in Cursor, Windows “Environment variables,” or a cohort setup doc. If you are not the person who should see secrets, do not try to set them; hand the machine to the right person and continue with installs only.

How to check you have keys in Windows (no terminal)

  1. Press the Windows key, type environment, open Edit the system environment variables → Environment variables…
  2. Under User or System variables, look for names the trainer uses (e.g. GEMINI_API_KEY, GROQ_API_KEY).
  3. If a variable should exist but is missing, that is a job for whoever owns the key—not something you guess.
Before weeks that use agents, deployment, and RAG: keys and logins need to be working. The trainer can give you a short test link in Chrome; if that fails, fix sign-ins before blaming the project code.

3. Install on Windows (use links and normal installers)

These steps assume you are on an administrator account, or the installer can ask for an admin password. Work in the order 1 → 8 the first time you set up a machine. You do not need a terminal: every tool has a normal Download for Windows button. A few “If you get stuck / for IT” notes mention optional short commands.

Tip for slow networks: you can open each “Download” on a faster office PC, save the .exe files to a shared drive or a USB drive, and run them in the lab—still the same wizards, just not downloaded twice per room.

1 · Google Chrome Required

Chrome is the browser we test with for the learning platform, quizzes, and most cloud tools. Install it before everything else so you can follow the rest of this page.

Official download: https://www.google.com/chrome/ — choose the normal Windows 64-bit installer (your PC is almost always 64-bit on modern i5s).

IT mass-deploy option (only if you use the college’s device management): Google Chrome for Business.

  1. Open the first link, click Download / Accept and install if asked.
  2. When the ChromeSetup.exe (or similar) download finishes, open it from the browser or from your Downloads folder and say Yes to the Windows question.
  3. When the install finishes, pin Chrome to the taskbar: open Chrome once, right‑click its icon on the taskbar, choose Pin to taskbar if you want.
  4. Set Chrome as the default for web links: Start → Settings → Apps → Default apps → pick a common type (for example, .html) and choose Chrome—exact wording can vary a bit on Windows 10 vs 11.

Quick “does it work” check (no code)

After install, the Chrome window should show the Google or “Welcome” page. In the top corner, the three dots menu (⋮) → Help → About Google Chrome should show a current version and say Chrome is up to date. That is enough.

2 · Cursor (main code editor) Required

Cursor is the primary editor for the workshop. Your trainer may send a specific invite, license, or team name—read their email in parallel with the steps below.

Download: https://www.cursor.com/downloads (pick Windows (x64) or “User installer” for Windows, per that page at install time.)

  1. Run the downloaded installer, accept the licence, and use the default folder unless your IT policy says different.
  2. On first open, if you see a sign-in screen, use the method your cohort agreed on (often a Google, GitHub, or Cursor sign-in; your trainer or IT admin may create accounts in advance.)
  3. Use File → Open Folder and try opening a simple new empty folder. If a chat/assistant side panel is visible and does not show an error, you are in good shape for class.
  4. If a corporate “SmartScreen” warning appears, click More info → Run anyway only if your team approved Cursor as software on lab PCs.

3 · Node.js (LTS) Required

Node is used in several lessons. Install the Long Term Support (LTS) line, not a preview.

Download: https://nodejs.org/en — the big green button should say something like 20.x LTS (version number can change; “LTS” is what you want.)

  1. Run the node-v…-x64.msi installer. Leave “Add to PATH” turned on if the wizard shows it (this lets other tools find Node without extra work).
  2. At the end, the wizard may offer to install Tools for native modules—on locked-down labs, skip unless your trainer specifically asked; you can add them later with IT if a lesson fails for that reason only.
  3. Reboot the PC once if the next lesson step says Node is still missing. Some campus policies need a new login session before PATH updates.
Optional: IT version check (one line)

Open Command Prompt and run: node -v and npm -v. A modern LTS will show a line starting with v20 or higher for node -v (as of 2026, check your lesson doc if a minimum is stricter.)

4 · Python 3.11+ Required

Python 3.11 (or a newer 3.11+ release) is used in multiple labs. The Windows “Microsoft Store” shortcut can be confusing; use the real installer from python.org unless IT gave you a different, approved build.

Download: https://www.python.org/downloads/windows/ — choose the latest 3.1x for Windows, 64-bit installer (often named “Windows installer (64-bit)” and ends in .exe).

  1. On the first installer screen, tick the box: Add python.exe to PATH. This is the step people miss most often.
  2. When asked, we recommend “Install for all users” on a shared machine if that option is shown.
  3. On the optional features page, the defaults are usually enough; the trainer can tell you if “pip” or “td/tk and IDLE” are required for a specific day.
  4. At the end, the installer may show “Disable path length limit”—click it if the button is available, then close the installer when it finishes.
Only if a lesson still says “python not found”

In Command Prompt try py -3.11 -V or python -V after sign-out and sign-in. If the py launcher is missing, repair the same installer from the Control Panel, or re-run the installer and choose Modify → make sure py launcher and PATH are on.

5 · Git for Windows Required

Git is how students copy, update, and turn in project folders. The installer is long, but the defaults on each screen are fine for a classroom: “Checkout Windows-style, commit Unix-style” is normal.

Download: https://git-scm.com/download/win (the page will auto-suggest a 64-bit Git-…-64-bit.exe for most PCs).

  1. Run the installer, click Next on each screen, and accept recommended defaults unless your IT has a one-page override.
  2. When the installer offers an editor, Use Visual Studio Code as Git’s default editor is a reasonable choice if you already put VS Code on; otherwise the minimal Vim-based option is still OK for command-line work only.
  3. On Adjusting your PATH, the common choice is Git from the command line and also from 3rd-party software—that keeps lessons simple.
  4. Reboot or sign out and back in if another tool still cannot find git after a successful install.

GitHub sign-in in the browser

Class does not use “mystical terminal login” in front of you if we can help it. Students usually sign in to github.com in Chrome first, then in Cursor/VS Code use the built-in “Clone from GitHub” and the browser to approve. Your trainer can demo this on Day 1 if needed.

6 · Ollama (local models) Required

Ollama runs on the PC to download and run local AI models (larger than normal apps; plan a quiet time on good Wi-Fi).

Download: https://ollama.com/download for Windows, then run the setup.exe or Microsoft Store path—follow whatever that page offers when you read it. Do this when you have a reliable connection and a few gigabytes of free space.

  1. Run the Ollama installer, accept defaults, and let it add itself to the Start menu.
  2. After it finishes, open Ollama from the Windows Start list at least once so the background service can start. If you see a small llama or icon in the notification area, that is normal.
  3. Go back to the workshop materials or your email and use the exact model names your trainer listed (we often use a small “chat” model and a small “embedding” model—names change; do not guess on your own).
Downloading models (usually needs one short “command” window per model)

Ollama’s public docs still expect ollama pull <model> for a reliable download. A lab IT person can: open Command Prompt (or PowerShell) as a normal user, and run, for example:

ollama pull llama3.2:3b
ollama pull nomic-embed-text

Replace the names with what your trainer wrote for this cohort. Wait for each to finish. Then run a test line the trainer provides (often one plain English message) to make sure the model is warm, not 10:00 a.m. on the first use.

Class morning: do the first model run on a break the day before, or right after install—not when students are walking in, unless the trainer has a very small demo model in mind.

7 · Visual Studio Code (optional backup) Optional

Cursor is the main tool. Some faculty keep VS Code on the same PC as a lightweight backup, or to match a college that already standardised on it.

Download: https://code.visualstudio.com/ — “Windows” user or system installer per your policy.

  1. Run the installer, and prefer system install / all users for shared rooms if that choice appears.
  2. Optional: on the “Select additional tasks” page, a desktop icon can help if students look for a blue “VS” logo.
  3. First launch, skip heavy theme/setup wizards; you can always tune later. Do not need to “sync settings” in the lab unless the college uses that.

8 · Docker Desktop (only if the cohort uses containers) Optional

Some advanced lessons use containers (isolated “mini-machines” for software). That is the only time you need Docker. If the cohort schedule has no container week, you can skip this entirely.

Start here, then follow the current Windows guide linked from that page: Docker Desktop and Windows install (official docs) — requirements change; your IT team should use this official path.

Most colleges: a network admin or IT desk installs WSL2 or Hyper-V and Docker, because a normal download may fail on locked systems. This block is a pointer, not a promise that a single .exe is enough in every school.

Need Mac/Linux, exact minimum versions, or a scripted one-shot installer? Gnanalytica and your IT team have a separate pack – this public page is intentionally the Windows, click friendly path.

4. Check everything works (no terminal required)

Walk this list on one lab user account the same way a student will log in. If something is red, return to the matching block in section 3 (or get IT) before the cohort starts. Exact version numbers are less important than “it opens, it signs in, it can run a tiny demo.”

IT or advanced staff only: if you prefer exact version output, see the Optional version commands at the end of this section. Normal lab setup does not need that.

Quick list (tick mentally or print the page with Ctrl + P)

VS Code and Docker (only if you installed them)

If you put VS Code on the machine, open it once, skip account sync until policy allows, and know where the blue “VS” icon is. If Docker is required for a later week, open Docker Desktop and wait until it is “green”/running, then the trainer/IT can run a hello-world. Otherwise skip both.

Optional: exact --version style checks (for IT)

Open Command Prompt and run, one at a time—compare approximations with your release plan (LTS for Node, Python 3.11+ for python, Ollama current release from the install):

node -v
py -3.11 -V
where git
ollama -v
ollama list
docker -v

On Windows, prefer py -3.11 -V if python -V is ambiguous. Chrome and Cursor are already confirmed via the normal UI.

5. Get help (escalation)

If a step in section 4 fails and a simple retry or reboot does not help, open a support thread with a short story of what you clicked and what the screen said. You do not need a terminal to escalate—use screenshots.

  • Windows build — Start → Settings → System → About and note the edition and version (for example, Windows 11, 23H2) or a photo of that screen
  • Which part failed — e.g. “Ollama — no tray icon after reboot” or “Cursor — sign-in loop”
  • Words on the error — full-screen snip, or the exact message text
  • Only for IT / optional: if a command from the grey “IT only” box was used, a screenshot of that Command Prompt window

Primary channel — Community Board

Post to board.html with the tag setup. This creates a searchable record and allows the full support team to respond. Other Support Faculty may have solved the same issue.

Same-day blockers

For issues that block a session starting in less than 2 hours, use your cohort Slack or WhatsApp group directly. Ping the trainer first — they have escalation paths to Gnanalytica infrastructure.
